### Finding 1 — Unnecessary `unsafe impl Send for NoopTx`

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/tests/publisher_mock_test.rs:54-56` |
| **Severity** | High |
| **Category** | Correctness |

**Problem**

`NoopTx` is a unit struct with no fields. Unit structs are automatically `Send + Sync` because they contain nothing that could break those bounds. The `unsafe impl Send` is therefore (a) redundant, (b) a code smell — `unsafe` is a load-bearing keyword and signalling it for a no-op trains the eye to skip past real `unsafe`. It also hides the fact that `Sync` is *not* implemented, which would matter the moment anyone tries to share a `NoopTx` across tasks.

**Context** (surrounding code as it exists today)

```rust
// crates/outbox-publisher/tests/publisher_mock_test.rs:53-56
// A dummy transaction type — no real DB needed for unit tests.
pub struct NoopTx;

unsafe impl Send for NoopTx {}
```

**Recommended fix**

```rust
// A dummy transaction type — no real DB needed for unit tests.
pub struct NoopTx;
```

**Why this fix**

Removing the `unsafe` keeps the test code idiomatic and the compiler infers `Send + Sync` automatically. If at any point `NoopTx` gains a non-`Send` field, the compiler will report it precisely instead of being silenced by a hand-rolled impl.

### Finding 2 — Missing `append_with_id` on the `Publisher` trait

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/src/publisher.rs:17-46` |
| **Severity** | High |
| **Category** | Correctness |

**Problem**

CLAUDE.md elevates caller-provided `event_id` to a “first-class API” (resolves TDD §10 Q2): `append_with_id` lets applications use deterministic UUIDs for cross-system idempotency. Phase 1 owns the *trait shape*; Phase 2 only adds the SQLx *implementation*. Shipping the trait without `append_with_id` forces a breaking change in Phase 2 to add the missing method (or to add a default impl that does the wrong thing). Even the rustdoc on `append` already references `append_with_id` (line 25), so its absence is a documented inconsistency.

**Context** (surrounding code as it exists today)

```rust
// crates/outbox-publisher/src/publisher.rs:23-46
    /// Append a single event to the outbox inside the given transaction.
    ///
    /// A UUID v4 `event_id` is generated internally. Use `append_with_id` if
    /// you need a deterministic identifier.
    fn append<'a, 'b, E>(
        &'a self,
        tx: &'b mut Self::Tx<'a>,
        event: &'b E,
        ctx: &'b EventContext,
    ) -> impl Future<Output = Result<EventId, PublishError>> + Send + 'b
    where
        E: DomainEvent + Serialize + Send + Sync,
        'a: 'b;

    /// Append multiple events to the outbox in a single round-trip.
    fn append_batch<'a, 'b, E>(
        &'a self,
        tx: &'b mut Self::Tx<'a>,
        events: &'b [(E, EventContext)],
    ) -> impl Future<Output = Result<Vec<EventId>, PublishError>> + Send + 'b
    where
        E: DomainEvent + Serialize + Send + Sync,
        'a: 'b;
```

**Recommended fix**

Add the method to the trait alongside `append` and `append_batch`:

```rust
    /// Append a single event using a caller-supplied `event_id`.
    ///
    /// Useful for deterministic idempotency keys (e.g. UUID v5 derived from
    /// the input). On unique-constraint violation the call returns
    /// [`PublishError::DuplicateEventId`].
    fn append_with_id<'a, 'b, E>(
        &'a self,
        tx: &'b mut Self::Tx<'a>,
        event_id: EventId,
        event: &'b E,
        ctx: &'b EventContext,
    ) -> impl Future<Output = Result<EventId, PublishError>> + Send + 'b
    where
        E: DomainEvent + Serialize + Send + Sync,
        'a: 'b;
```

Also update the hand-rolled mock in `tests/publisher_mock_test.rs` to implement it.

**Why this fix**

Putting the method on the trait now (Phase 1) locks the API contract before any adapter exists, so Phase 2 only has to write the SQLx body. Postponing it means either a breaking trait change later or every adapter author having to re-discover the convention.

### Finding 3 — `PublishError::Database(String)` discards the underlying error and lacks a unique-violation variant

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/src/error.rs:1-10` |
| **Severity** | Medium |
| **Category** | Correctness |

**Problem**

Two issues:

1. `Database(String)` collapses every driver-level error into a `String`. The caller cannot programmatically distinguish a deadlock from a constraint violation from a connection drop, and the chain of `source()` errors is lost. Phase 2 will need `#[from] sqlx::Error` (or a driver-agnostic enum) to be useful.
2. The checklist (and the `append_with_id` contract from TDD §10 Q2) requires that unique-constraint violations on caller-provided IDs surface as a *distinguishable* variant — currently there is no `DuplicateEventId`.

Decoupling now avoids forcing every consumer to string-match in Phase 2.

**Context** (surrounding code as it exists today)

```rust
// crates/outbox-publisher/src/error.rs:1-10
/// Errors that can occur when publishing events to the outbox.
#[derive(Debug, thiserror::Error)]
pub enum PublishError {
    /// A database-level error occurred while inserting the outbox row.
    #[error("database error: {0}")]
    Database(String),
    /// The event payload could not be serialized to JSON.
    #[error("payload serialization failed: {0}")]
    Serialization(#[from] serde_json::Error),
}
```

**Recommended fix**

The umbrella crate stays driver-agnostic, so use an opaque source error rather than a hard dependency on `sqlx::Error`. The SQLx adapter will provide the `From<sqlx::Error>` impl on its side.

```rust
/// Errors that can occur when publishing events to the outbox.
#[derive(Debug, thiserror::Error)]
pub enum PublishError {
    /// The caller-supplied `event_id` already exists in the outbox.
    #[error("duplicate event id")]
    DuplicateEventId,
    /// A database-level error occurred while inserting the outbox row.
    #[error("database error: {0}")]
    Database(#[source] Box<dyn std::error::Error + Send + Sync>),
    /// The event payload could not be serialized to JSON.
    #[error("payload serialization failed: {0}")]
    Serialization(#[from] serde_json::Error),
}
```

**Why this fix**

`Box<dyn Error + Send + Sync>` preserves the original error chain (`source()` traversal still works) without leaking `sqlx` into the umbrella crate. `DuplicateEventId` is the variant `append_with_id` callers need to handle for idempotency.

**Recommended fix**

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]

    /// Return the inner [`Uuid`].

**Recommended fix**

Drop the unused tuple element:

```rust
fn parse_struct_attrs(ast: &DeriveInput) -> syn::Result<(String, String)> {
    let mut kind: Option<String> = None;
    let mut aggregate: Option<String> = None;

    for attr in &ast.attrs {
        if !attr.path().is_ident("event") {
            continue;
        }

        attr.parse_nested_meta(|meta| {
            if meta.path.is_ident("kind") {
                let value = meta.value()?;
                let lit: Lit = value.parse()?;
                if let Lit::Str(s) = &lit {
                    if kind.is_some() {
                        return Err(meta.error("`kind` specified more than once"));
                    }
                    kind = Some(s.value());
                } else {
                    return Err(meta.error("`kind` must be a string literal"));
                }
            } else if meta.path.is_ident("aggregate") {
                let value = meta.value()?;
                let lit: Lit = value.parse()?;
                if let Lit::Str(s) = &lit {
                    if aggregate.is_some() {
                        return Err(meta.error("`aggregate` specified more than once"));
                    }
                    aggregate = Some(s.value());
                } else {
                    return Err(meta.error("`aggregate` must be a string literal"));
                }
            } else if meta.path.is_ident("aggregate_id") {
                return Err(meta.error("`aggregate_id` is a field attribute, not a struct attribute"));
            } else {
                return Err(meta.error("unknown `#[event]` attribute key"));
            }
            Ok(())
        })?;
    }

    let kind_val = kind.ok_or_else(|| syn::Error::new_spanned(
        &ast.ident,
        "missing required `#[event(kind = \"...\")]` attribute",
    ))?;
    let aggregate_val = aggregate.ok_or_else(|| syn::Error::new_spanned(
        &ast.ident,
        "missing required `#[event(aggregate = \"...\")]` attribute",
    ))?;

    Ok((kind_val, aggregate_val))
}
```

**Why this fix**

Removing dead state shrinks the function and prevents readers from chasing what `span_ts` is for. Replacing the matches with `ok_or_else` is the idiomatic `Option` handling per CLAUDE.md.

### Finding 13 — Tokio version constraint is `>=1.20` instead of a caret range

| Field | Value |
|-------|-------|
| **File:Line** | `Cargo.toml:27` |
| **Severity** | Low |

Cargo’s default semver behaviour (`"1"` or `"1.20"`) would allow any 1.x ≥ the minimum but block 2.x. # Cargo.toml:27

```toml
tokio = { version = "1.20", features = ["rt-multi-thread", "macros"] }
```

…or move the heavier features (`fs`, `net`, `signal`, etc.) into the dev-dependency override of crates that actually need them.

**Why this fix**

Caret ranges (`"1.20"`) are the idiomatic Rust expression of “1.20 or any compatible later 1.x”. Narrowing `features` reduces compile time and shrinks the dependency closure.

### Finding 15 — Mock test uses `std::sync::Mutex` instead of `tokio::sync::Mutex`

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/tests/publisher_mock_test.rs:38, 45, 50, 72, 88` |
| **Severity** | Low |
| **Category** | Concurrency |

**Problem**

CLAUDE.md says: *“Use `tokio::sync` primitives in async code, not `std::sync::Mutex` / `RwLock`.”* This test holds `std::sync::Mutex` inside `#[tokio::test]` `async fn`s. In practice the lock is only held for sync `push`/`clone` (no `.await` while held) so it’s safe — but the rule exists so contributors don’t have to audit each callsite. A test that demonstrates the trait is also a template downstream consumers will copy.

**Context** (surrounding code as it exists today)

```rust

struct RecordingPublisher {
    appended: std::sync::Mutex<Vec<String>>,

impl RecordingPublisher {
    fn new() -> Self {
        Self {
            appended: std::sync::Mutex::new(Vec::new()),
        }
    }
```

**Recommended fix**

Since this is sync state used inside an async test, `std::sync::Mutex` is technically the *right* tool (no async, no contention). The fix is either:

- replace with `tokio::sync::Mutex` and `.lock().await` to match the convention, or
- relax the CLAUDE.md rule with an exception: *“…unless the critical section is purely synchronous and never crosses `.await`.”* and add a one-line `// sync critical section only` comment here.

Recommended: relax the rule (option 2) — `tokio::sync::Mutex` is genuinely slower than `std::sync::Mutex` for purely-sync state.

**Why this fix**

Either path keeps the example honest. The first matches the literal rule; the second is what most production codebases settle on once they realise `tokio::sync::Mutex` has a non-trivial cost.

### Finding 19 — `EventContext::with_metadata` doc claims a constraint it never enforces

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/src/event.rs:95-99` |
| **Severity** | Low |
| **Category** | Correctness |

**Problem**

The rustdoc on `with_metadata` says: *“Replace the metadata value (must be a JSON object or `null`).”* The function takes `serde_json::Value` and stores it verbatim — passing `Value::Array(...)`, `Value::String("...")`, or any other variant is silently accepted. Phase 2 will write this to the `outbox_events.metadata` column, which is `JSONB` — Postgres will accept any JSON, but downstream consumers (the dispatcher’s webhook envelope) may assume the field is always an object. Either the doc is aspirational and should be enforced, or the constraint is wrong and the doc should drop it.

**Context**

```rust
// crates/outbox-publisher/src/event.rs:95-99
/// Replace the metadata value (must be a JSON object or `null`).
pub fn with_metadata(mut self, metadata: serde_json::Value) -> Self {
    self.metadata = metadata;
    self
}
```

The default constructor at line 71 produces `Value::Object(Map::new())`, suggesting the intent really is “object-or-null”.

**Recommended fix**

Two options:

1. **Enforce.** Change the signature to `fn with_metadata(mut self, metadata: serde_json::Map<String, serde_json::Value>) -> Self` and store `Value::Object(metadata)`. This is the type-system-level guarantee.
2. **Document but don’t enforce.** Drop “must be” language; reword to *“Replace the metadata value. Production usage typically passes a JSON object; arrays and scalars are accepted but may not round-trip through the dispatcher’s webhook envelope.”*

Option 1 is preferable for a library API.

**Why this fix**

A doc that promises a constraint the code never checks is worse than no doc — readers internalise the rule, then are surprised when an array slips through. Phase 2/3 cross-language tests are easier to debug if the publisher refuses bad shapes at the boundary.

### Finding 21 — Duplicate `#[event(aggregate_id)]` on the same field is reported with a confusing span

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher-derive/src/lib.rs:178-202` |
| **Severity** | Low |
| **Category** | Proc-macro |

**Problem**

The inner `for attr in &field.attrs` loop pushes `(field, ident)` into `found` once per matching `#[event(aggregate_id)]`. If a user writes:

```rust
#[derive(DomainEvent)]
#[event(kind = "x@v1", aggregate = "x")]
struct X {
    #[event(aggregate_id)]
    #[event(aggregate_id)]
    id: Uuid,
}
```

`found` ends up with two entries pointing at the *same* field. The “multiple fields marked” branch (line 214) fires, but the diagnostic span points at `id` — implying the user has two fields with the marker, which is false. The actual bug is the duplicated *attribute* on one field.

This is a corner case; the impact is a less-helpful error message rather than a wrong compile.

**Context**

```rust
// crates/outbox-publisher-derive/src/lib.rs:178-202
for field in &named.named {
    for attr in &field.attrs {
        if !attr.path().is_ident("event") {
            continue;
        }
        let mut is_aggregate_id = false;
        attr.parse_nested_meta(|meta| { /* ... */ })?;
        if is_aggregate_id {
            let ident = field.ident.as_ref().expect("...");
            found.push((field, quote!(#ident)));   // can push the same field twice
        }
    }
}
```

**Recommended fix**

Track membership per field, or detect duplicate-attribute-on-same-field and emit a precise diagnostic:

```rust
for field in &named.named {
    let mut field_marked = false;
    for attr in &field.attrs {
        if !attr.path().is_ident("event") {
            continue;
        }
        let mut is_aggregate_id = false;
        attr.parse_nested_meta(|meta| { /* unchanged */ })?;
        if is_aggregate_id {
            if field_marked {
                return Err(syn::Error::new_spanned(
                    attr,
                    "`#[event(aggregate_id)]` specified more than once on the same field",
                ));
            }
            field_marked = true;
            let ident = field.ident.as_ref().expect("Fields::Named guarantees field.ident");
            found.push((field, quote!(#ident)));
        }
    }
}
```

**Why this fix**

A precise diagnostic is the proc-macro’s only contract; a misleading span is the kind of thing that wastes a user 10 minutes once they see it.

### Finding 1 — Unnecessary `unsafe impl Send for NoopTx`

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/tests/publisher_mock_test.rs:54-56` |
| **Severity** | High |
| **Category** | Correctness |

**Problem**

`NoopTx` is a unit struct with no fields. Unit structs are automatically `Send + Sync` because they contain nothing that could break those bounds. The `unsafe impl Send` is therefore (a) redundant, (b) a code smell — `unsafe` is a load-bearing keyword and signalling it for a no-op trains the eye to skip past real `unsafe`. It also hides the fact that `Sync` is *not* implemented, which would matter the moment anyone tries to share a `NoopTx` across tasks.

**Context** (surrounding code as it exists today)

```rust
// crates/outbox-publisher/tests/publisher_mock_test.rs:53-56
// A dummy transaction type — no real DB needed for unit tests.
pub struct NoopTx;

unsafe impl Send for NoopTx {}
```

**Recommended fix**

```rust
// A dummy transaction type — no real DB needed for unit tests.
pub struct NoopTx;
```

**Why this fix**

Removing the `unsafe` keeps the test code idiomatic and the compiler infers `Send + Sync` automatically. If at any point `NoopTx` gains a non-`Send` field, the compiler will report it precisely instead of being silenced by a hand-rolled impl.

### Finding 2 — Missing `append_with_id` on the `Publisher` trait

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/src/publisher.rs:17-46` |
| **Severity** | High |
| **Category** | Correctness |

**Problem**

CLAUDE.md elevates caller-provided `event_id` to a “first-class API” (resolves TDD §10 Q2): `append_with_id` lets applications use deterministic UUIDs for cross-system idempotency. Phase 1 owns the *trait shape*; Phase 2 only adds the SQLx *implementation*. Shipping the trait without `append_with_id` forces a breaking change in Phase 2 to add the missing method (or to add a default impl that does the wrong thing). Even the rustdoc on `append` already references `append_with_id` (line 25), so its absence is a documented inconsistency.

**Context** (surrounding code as it exists today)

```rust
// crates/outbox-publisher/src/publisher.rs:23-46
    /// Append a single event to the outbox inside the given transaction.
    ///
    /// A UUID v4 `event_id` is generated internally. Use `append_with_id` if
    /// you need a deterministic identifier.
    fn append<'a, 'b, E>(
        &'a self,
        tx: &'b mut Self::Tx<'a>,
        event: &'b E,
        ctx: &'b EventContext,
    ) -> impl Future<Output = Result<EventId, PublishError>> + Send + 'b
    where
        E: DomainEvent + Serialize + Send + Sync,
        'a: 'b;

    /// Append multiple events to the outbox in a single round-trip.
    fn append_batch<'a, 'b, E>(
        &'a self,
        tx: &'b mut Self::Tx<'a>,
        events: &'b [(E, EventContext)],
    ) -> impl Future<Output = Result<Vec<EventId>, PublishError>> + Send + 'b
    where
        E: DomainEvent + Serialize + Send + Sync,
        'a: 'b;
```

**Recommended fix**

Add the method to the trait alongside `append` and `append_batch`:

```rust
    /// Append a single event using a caller-supplied `event_id`.
    ///
    /// Useful for deterministic idempotency keys (e.g. UUID v5 derived from
    /// the input). On unique-constraint violation the call returns
    /// [`PublishError::DuplicateEventId`].
    fn append_with_id<'a, 'b, E>(
        &'a self,
        tx: &'b mut Self::Tx<'a>,
        event_id: EventId,
        event: &'b E,
        ctx: &'b EventContext,
    ) -> impl Future<Output = Result<EventId, PublishError>> + Send + 'b
    where
        E: DomainEvent + Serialize + Send + Sync,
        'a: 'b;
```

Also update the hand-rolled mock in `tests/publisher_mock_test.rs` to implement it.

**Why this fix**

Putting the method on the trait now (Phase 1) locks the API contract before any adapter exists, so Phase 2 only has to write the SQLx body. Postponing it means either a breaking trait change later or every adapter author having to re-discover the convention.

### Finding 3 — `PublishError::Database(String)` discards the underlying error and lacks a unique-violation variant

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/src/error.rs:1-10` |
| **Severity** | Medium |
| **Category** | Correctness |

**Problem**

Two issues:

1. `Database(String)` collapses every driver-level error into a `String`. The caller cannot programmatically distinguish a deadlock from a constraint violation from a connection drop, and the chain of `source()` errors is lost. Phase 2 will need `#[from] sqlx::Error` (or a driver-agnostic enum) to be useful.
2. The checklist (and the `append_with_id` contract from TDD §10 Q2) requires that unique-constraint violations on caller-provided IDs surface as a *distinguishable* variant — currently there is no `DuplicateEventId`.

Decoupling now avoids forcing every consumer to string-match in Phase 2.

**Context** (surrounding code as it exists today)

```rust
// crates/outbox-publisher/src/error.rs:1-10
/// Errors that can occur when publishing events to the outbox.
#[derive(Debug, thiserror::Error)]
pub enum PublishError {
    /// A database-level error occurred while inserting the outbox row.
    #[error("database error: {0}")]
    Database(String),
    /// The event payload could not be serialized to JSON.
    #[error("payload serialization failed: {0}")]
    Serialization(#[from] serde_json::Error),
}
```

**Recommended fix**

The umbrella crate stays driver-agnostic, so use an opaque source error rather than a hard dependency on `sqlx::Error`. The SQLx adapter will provide the `From<sqlx::Error>` impl on its side.

```rust
/// Errors that can occur when publishing events to the outbox.
#[derive(Debug, thiserror::Error)]
pub enum PublishError {
    /// The caller-supplied `event_id` already exists in the outbox.
    #[error("duplicate event id")]
    DuplicateEventId,
    /// A database-level error occurred while inserting the outbox row.
    #[error("database error: {0}")]
    Database(#[source] Box<dyn std::error::Error + Send + Sync>),
    /// The event payload could not be serialized to JSON.
    #[error("payload serialization failed: {0}")]
    Serialization(#[from] serde_json::Error),
}
```

**Why this fix**

`Box<dyn Error + Send + Sync>` preserves the original error chain (`source()` traversal still works) without leaking `sqlx` into the umbrella crate. `DuplicateEventId` is the variant `append_with_id` callers need to handle for idempotency.

**Recommended fix**

Drop the unused tuple element:

```rust
fn parse_struct_attrs(ast: &DeriveInput) -> syn::Result<(String, String)> {
    let mut kind: Option<String> = None;
    let mut aggregate: Option<String> = None;

    for attr in &ast.attrs {
        if !attr.path().is_ident("event") {
            continue;
        }

        attr.parse_nested_meta(|meta| {
            if meta.path.is_ident("kind") {
                let value = meta.value()?;
                let lit: Lit = value.parse()?;
                if let Lit::Str(s) = &lit {
                    if kind.is_some() {
                        return Err(meta.error("`kind` specified more than once"));
                    }
                    kind = Some(s.value());
                } else {
                    return Err(meta.error("`kind` must be a string literal"));
                }
            } else if meta.path.is_ident("aggregate") {
                let value = meta.value()?;
                let lit: Lit = value.parse()?;
                if let Lit::Str(s) = &lit {
                    if aggregate.is_some() {
                        return Err(meta.error("`aggregate` specified more than once"));
                    }
                    aggregate = Some(s.value());
                } else {
                    return Err(meta.error("`aggregate` must be a string literal"));
                }
            } else if meta.path.is_ident("aggregate_id") {
                return Err(meta.error("`aggregate_id` is a field attribute, not a struct attribute"));
            } else {
                return Err(meta.error("unknown `#[event]` attribute key"));
            }
            Ok(())
        })?;
    }

    let kind_val = kind.ok_or_else(|| syn::Error::new_spanned(
        &ast.ident,
        "missing required `#[event(kind = \"...\")]` attribute",
    ))?;
    let aggregate_val = aggregate.ok_or_else(|| syn::Error::new_spanned(
        &ast.ident,
        "missing required `#[event(aggregate = \"...\")]` attribute",
    ))?;

    Ok((kind_val, aggregate_val))
}
```

**Why this fix**

Removing dead state shrinks the function and prevents readers from chasing what `span_ts` is for. Replacing the matches with `ok_or_else` is the idiomatic `Option` handling per CLAUDE.md.

### Finding 13 — Tokio version constraint is `>=1.20` instead of a caret range

| Field | Value |
|-------|-------|
| **File:Line** | `Cargo.toml:27` |

Cargo’s default semver behaviour (`"1"` or `"1.20"`) would allow any 1.x ≥ the minimum but block 2.x.
```toml
tokio = { version = "1.20", features = ["rt-multi-thread", "macros"] }
```

…or move the heavier features (`fs`, `net`, `signal`, etc.) into the dev-dependency override of crates that actually need them.

**Why this fix**

Caret ranges (`"1.20"`) are the idiomatic Rust expression of “1.20 or any compatible later 1.x”. Narrowing `features` reduces compile time and shrinks the dependency closure.

### Finding 23 — `is_uuid_type` accepts any `X::uuid::Uuid` path, not just the real crate

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher-derive/src/lib.rs:247-256` |
| **Severity** | Low |
| **Category** | Proc-macro |

**Problem**

The `[.., prev, last]` arm matches any two-segment suffix where the penultimate segment is literally `uuid` and the last is `Uuid`, regardless of the leading path. A user-defined type `mymod::uuid::Uuid` (3 segments: `mymod`, `uuid`, `Uuid`) passes the check — the `..` wildcard absorbs `mymod`. The macro then generates `fn aggregate_id(&self) -> ::uuid::Uuid { self.field }`, which fails to compile with a confusing "expected `uuid::Uuid`, found a different `uuid::Uuid`" message pointing at the `#[derive]` line rather than the field declaration.

In practice, a user creating a `uuid` sub-module whose inner `Uuid` collides with this pattern is unusual, but the diagnostic is worse than it would be without the `[..]` widening.

**Context**

```rust
// crates/outbox-publisher-derive/src/lib.rs:247-256
fn is_uuid_type(ty: &Type) -> bool {
    let Type::Path(TypePath { qself: None, path }) = ty else {
        return false;
    };
    let segs: Vec<&syn::PathSegment> = path.segments.iter().collect();
    match segs.as_slice() {
        [last] => last.ident == "Uuid",
        [.., prev, last] => last.ident == "Uuid" && prev.ident == "uuid",
        _ => false,
    }
}
```

**Recommended fix**

Tighten the multi-segment arm to at most three segments (handling `::uuid::Uuid` as two `path.segments` entries because the leading `::` is encoded in `path.leading_colon`, not as an extra segment):

```rust
fn is_uuid_type(ty: &Type) -> bool {
    let Type::Path(TypePath { qself: None, path }) = ty else {
        return false;
    };
    let segs: Vec<&syn::PathSegment> = path.segments.iter().collect();
    match segs.as_slice() {
        // `Uuid`
        [only] => only.ident == "Uuid",
        // `uuid::Uuid` or `::uuid::Uuid` (leading_colon is on the Path, not a segment)
        [crate_seg, ty_seg] => crate_seg.ident == "uuid" && ty_seg.ident == "Uuid",
        _ => false,
    }
}
```

`path.leading_colon` being `Some` vs `None` distinguishes `uuid::Uuid` from `::uuid::Uuid`; neither adds an extra segment, so the two-element arm handles both.

**Why this fix**

Restricting to exactly one or two segments closes the false-positive window. Any field written as `mymod::uuid::Uuid` is rejected with a clear "must be of type `Uuid`" span on the field type, rather than a compile error on the `#[derive]` line after the macro has already decided the type is valid.

### Finding 25 — `EventContext::metadata()` returns `&serde_json::Value`, dropping the invariant the setter enforces

|-------|-------|
| **File:Line** | `crates/outbox-publisher/src/event.rs:93-95` |

```rust
            .as_object()
```

### Finding 30 — Hand-rolled mock records side effects synchronously, not inside the returned future

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/tests/publisher_mock_test.rs:73-74, 89-90, 103-110` |
| **Severity** | Medium |
| **Category** | Correctness |

**Problem**

`RecordingPublisher::append`, `append_with_id`, and `append_batch` all push into `self.appended` *before* the `async {}` block they return — i.e. on the synchronous call path that constructs the future. Concretely:

```rust

{

    async { Ok(EventId::from(Uuid::new_v4())) }   // <-- future yet to be awaited
}
```

Two consequences:

1. **Behaviour mismatch with any real `Publisher` implementation.** A SQLx adapter does its DB work *inside* the future (it has to — the `INSERT` is `.await`-driven). A test against this mock cannot catch a bug where production code constructs the future, drops it without polling, and silently skips the insert. Calling `let _ = publisher.append(...);` (no `.await`) on the mock still records the event; on the real impl it does nothing.
2. **CLAUDE.md elevates this file to “the reference pattern” for downstream consumers** (*“See `tests/publisher_mock_test.rs` for the reference pattern”*, updated for Finding 4). Every external user copying the file inherits the same shape.

### Finding 35 — `PublishError::Database` displays the source twice when consumers walk the chain

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/src/error.rs:7-9` |
| **Severity** | Low |
| **Category** | Correctness |

**Problem**

The `Database` variant uses `{0}` in its `Display` template **and** marks the field `#[source]`:

```rust
#[error("database error: {0}")]
Database(#[source] Box<dyn std::error::Error + Send + Sync>),
```

`{0}` interpolates the inner error’s `Display`, so `PublishError::Database(...).to_string()` emits `"database error: connection refused"`. Because the field is *also* `#[source]`, `err.source()` returns the same inner error. Any downstream code that walks the chain (e.g. `anyhow::Error::chain`, `tracing-error::ErrorLayer`, or the recommended `loop { ... err.source() }` pattern from the Rust API guidelines) will print the inner `Display` a second time, producing:

```text
database error: connection refused
caused by: connection refused
```

This is a known thiserror gotcha. The fix is to pick *one* role for the inner error: either it’s the `Display` payload (drop `#[source]`, but then `err.source()` returns `None` and chain-walking tools lose the inner type), or it’s the source (drop `{0}`, let chain-walkers render the inner). The latter is the convention the standard library and `anyhow` follow.

**Context** (current state)

```rust
// crates/outbox-publisher/src/error.rs:7-9
/// A database-level error occurred while inserting the outbox row.
#[error("database error: {0}")]
Database(#[source] Box<dyn std::error::Error + Send + Sync>),
```

The unit test at `error.rs:48-53` documents the duplicate behaviour:

```rust
let cause: Box<dyn std::error::Error + Send + Sync> =
    Box::new(std::io::Error::other("connection refused"));
let err = PublishError::Database(cause);
assert_eq!(err.to_string(), "database error: connection refused");
// And err.source().unwrap().to_string() also returns "connection refused".
```

**Recommended fix**

Drop the `{0}` interpolation; let the inner error render through `source()` traversal:

```rust
/// A database-level error occurred while inserting the outbox row.
#[error("database error")]
Database(#[source] Box<dyn std::error::Error + Send + Sync>),
```

…and update the test:

```rust
#[test]
fn publish_error_database_carries_source() {
    let cause: Box<dyn std::error::Error + Send + Sync> =
        Box::new(std::io::Error::other("connection refused"));
    let err = PublishError::Database(cause);
    assert_eq!(err.to_string(), "database error");
    assert_eq!(
        err.source().unwrap().to_string(),
        "connection refused",
    );
}
```

**Why this fix**

`PublishError::Database` is a wrapper over a driver-specific error. The job of the wrapper’s `Display` is to identify the *category* of failure; the job of the source is to carry the underlying detail. Mixing the two means every contextual logger duplicates the message, which is exactly the noise users tend to file as a bug.

### Finding 36 — Hand-rolled `EventContext::Default` where `#[derive(Default)]` would suffice

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/src/event.rs:71-80` |
| **Severity** | Low |
| **Category** | Idiom |

**Problem**

The hand-rolled `Default` produces exactly what `#[derive(Default)]` would produce — `Option::None` for the three IDs and `serde_json::Map::new()` for `metadata` (which is what `Map::default()` returns). Twelve lines of code where one attribute would do.

**Context** (current state)

```rust
// crates/outbox-publisher/src/event.rs:71-80
impl Default for EventContext {
    fn default() -> Self {
        Self {
            actor_id: None,
            correlation_id: None,
            causation_id: None,
            metadata: serde_json::Map::new(),
        }
    }
}
```

**Recommended fix**

```rust
#[derive(Debug, Clone, Default)]
pub struct EventContext {
    actor_id: Option<Uuid>,
    correlation_id: Option<Uuid>,
    causation_id: Option<Uuid>,
    metadata: serde_json::Map<String, serde_json::Value>,
}
```

…and delete the hand-rolled `impl Default`. The doctest assertion at `event_context_default_has_empty_metadata` keeps the behaviour pinned.

**Why this fix**

Less code to read, less to maintain, identical semantics. The hand-rolled impl was likely a holdover from when `metadata` was typed as `serde_json::Value` (Finding 25), where `Value::default()` would have produced `Value::Null` instead of an empty object. With the typed-`Map` migration the explicit impl no longer earns its keep.

### Finding 37 — `axum` feature is declared but activates nothing

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/Cargo.toml:15` |
| **Severity** | Low |
| **Category** | Idiom |

**Problem**

`axum = []` is an empty feature flag. It pulls in no dep, gates no code (`webhook/mod.rs` is a one-line stub), and adds no public symbols. A consumer enabling it gets the same crate they would have gotten without it. Phase 3 will eventually wire the axum extractor behind it, but until then the feature is part of the public API surface (`cargo add outbox-publisher --features axum` resolves) without any contract.

This is mostly harmless — `axum` is a name reservation — but the `derive` feature next to it actually does something (`dep:outbox-publisher-derive`), so the asymmetry invites confusion. A reader skimming `[features]` cannot tell which entries are real and which are pre-declared placeholders.

**Context** (current state)

```toml
# crates/outbox-publisher/Cargo.toml:12-15
[features]
default = []
derive = ["dep:outbox-publisher-derive"]
axum = []
```

**Recommended fix**

Either remove `axum` until Phase 3 lands, or annotate it as a placeholder so reviewers (and `cargo doc`) know it’s deliberate:

```toml
[features]
default = []
derive = ["dep:outbox-publisher-derive"]
# axum: reserved for the Phase 3 axum extractor; activates no code yet.
axum = []
```

The cleanest path is to add the feature *together* with the dep that activates it — i.e. introduce both in the Phase 3 PR. That keeps `cargo metadata` and `docs.rs` honest about what each feature buys.

**Why this fix**

Empty features that exist purely to reserve names tend to leak into ecosystem-wide tooling (e.g. `cargo-feature-set`, `cargo hack`) without producing useful build matrices. If the project really wants the name reserved, the comment makes it obvious; if it doesn’t, removing it now means there’s no semver concern when Phase 3 introduces the real one.

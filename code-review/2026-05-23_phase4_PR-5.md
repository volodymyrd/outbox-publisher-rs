### Finding 1 — Publish order is reversed: `outbox-publisher-sqlx` depends on `outbox-publisher`

| Field | Value |
|-------|-------|
| **File:Line** | `.github/workflows/ci.yml:91-99`, `.github/workflows/release.yml:36-57` |
| **Severity** | Critical |
| **Category** | Correctness |

**Problem**

Both workflows publish in the order `outbox-publisher-derive → outbox-publisher-sqlx → outbox-publisher`. But `outbox-publisher-sqlx` has `outbox-publisher.workspace = true` as a required (non-optional) dependency. The umbrella crate `outbox-publisher` must be on crates.io *before* `outbox-publisher-sqlx` is published — otherwise:

1. **`publish-dry-run` job**: the `cargo publish --dry-run -p outbox-publisher-sqlx` step runs verification (it does *not* use `--no-verify`), which builds the packaged tarball with path deps stripped. Since `outbox-publisher = "0.1.0"` is not on crates.io, the build will fail on the very first release attempt.
2. **`release.yml` publish job**: even with `--no-verify` the upload succeeds, but it leaves crates.io in a broken state — anyone who runs `cargo add outbox-publisher-sqlx` during the 20-second `sleep` window gets a package whose `outbox-publisher` dep cannot be resolved.

The correct dep-graph order is `outbox-publisher-derive → outbox-publisher → outbox-publisher-sqlx`.

**Context** (`.github/workflows/ci.yml:91-99`)

```yaml
      # Publish in dependency order: derive → sqlx adapter → umbrella.
      - name: cargo publish --dry-run (outbox-publisher-derive)
        run: cargo publish --dry-run -p outbox-publisher-derive

      - name: cargo publish --dry-run (outbox-publisher-sqlx)
        run: cargo publish --dry-run -p outbox-publisher-sqlx

      - name: cargo publish --dry-run (outbox-publisher)
        run: cargo publish --dry-run -p outbox-publisher
```

**Recommended fix** — `.github/workflows/ci.yml`

```yaml
      # Publish in dependency order: derive → umbrella → sqlx adapter.
      # outbox-publisher-sqlx depends on outbox-publisher, so the umbrella crate
      # must reach crates.io first.
      - name: cargo publish --dry-run (outbox-publisher-derive)
        run: cargo publish --dry-run -p outbox-publisher-derive

      - name: cargo publish --dry-run (outbox-publisher)
        run: cargo publish --dry-run -p outbox-publisher --all-features

      - name: cargo publish --dry-run (outbox-publisher-sqlx)
        run: cargo publish --dry-run -p outbox-publisher-sqlx
```

And `.github/workflows/release.yml` (mirror the same reordering — publish `outbox-publisher` before `outbox-publisher-sqlx`).

**Why this fix**

The publish topological order must match the dependency graph: a downstream crate can only be published once every upstream it pulls from crates.io is already there. `--all-features` on the dry-run for the umbrella crate also forces validation of the optional `outbox-publisher-derive` dep, catching the same class of bug for the derive feature path.

### Finding 2 — CI does not run doctests despite `#![deny(missing_docs)]` and rich rustdoc examples

| Field | Value |
|-------|-------|
| **File:Line** | `.github/workflows/ci.yml:55-56` |
| **Severity** | High |
| **Category** | Testing |

**Problem**

Phase 4 added `#![deny(missing_docs)]` and many code samples in rustdoc (e.g. `crates/outbox-publisher/src/lib.rs:16-42`, `crates/outbox-publisher-sqlx/src/lib.rs:6-35`, `crates/outbox-publisher/src/webhook/mod.rs:5-20`, `crates/outbox-publisher-derive/src/lib.rs:36-54`). cargo test --workspace --test '*'                    # integration tests only

`--lib` excludes doctests; `--test '*'` only matches `tests/*.rs` integration crates. Doctests are never executed — they will silently rot the moment an API rename breaks an example.

**Context** (`.github/workflows/ci.yml:42-75`)

```yaml

    runs-on: ubuntu-latest
    needs: check
    steps:
      - uses: actions/checkout@v4
      - name: Install Rust toolchain
        uses: dtolnay/rust-toolchain@stable
      - name: Cache Cargo registry + build artifacts
        uses: Swatinem/rust-cache@v2
      - name: cargo test (lib + unit, all features)
        run: cargo test --workspace --lib --all-features

  test-integration:
    name: Integration Tests
    ...
      - name: cargo test (integration)
        run: cargo test --workspace --test '*'
```

**Recommended fix**

```yaml
  test-unit:
    name: Unit & Doc Tests
    runs-on: ubuntu-latest
    needs: check
    steps:
      - uses: actions/checkout@v4
      - name: Install Rust toolchain
        uses: dtolnay/rust-toolchain@stable
      - name: Cache Cargo registry + build artifacts
        uses: Swatinem/rust-cache@v2
      - name: cargo test (lib + unit, all features)
        run: cargo test --workspace --lib --all-features
      - name: cargo test (doctests, all features)
        run: cargo test --workspace --doc --all-features
```

**Why this fix**

`--doc` is the only flag that compiles and runs doctest snippets. Splitting it into its own step (rather than dropping `--lib`) keeps failures legible and lets the doctest step be re-run independently.

### Finding 3 — CI toolchain (`@stable`) drifts from the repo's pinned `rust-toolchain.toml` (1.88.0)

| Field | Value |
|-------|-------|
| **File:Line** | `.github/workflows/ci.yml:24-27`, `.github/workflows/release.yml:28-29` |
| **Severity** | High |
| **Category** | Correctness |

**Problem**

`rust-toolchain.toml` pins `channel = "1.88.0"`. `CLAUDE.md` explicitly says this is "pinned to 1.88.0 in lockstep with outbox-dispatcher". But every CI job runs `dtolnay/rust-toolchain@stable`, which installs whatever the current stable is (e.g. 1.95.0). The rustup proxies in the runner will then auto-install 1.88.0 the first time `cargo` is invoked (because of `rust-toolchain.toml`) — but two consequences:

1. The `components: rustfmt, clippy` from the action install for *stable*, not 1.88.0. When `cargo fmt`/`cargo clippy` runs under 1.88.0 it will fall back to the 1.88.0 versions, downloading them on demand — sometimes silently using a different rustfmt edition / clippy lint set than locally.
2. Reviewers reading the CI assume the test matrix matches the local pin; it doesn't. The lockstep guarantee is broken in CI.

**Context** (`.github/workflows/ci.yml:22-27`)

```yaml
      - uses: actions/checkout@v4

      - name: Install Rust toolchain
        uses: dtolnay/rust-toolchain@stable
        with:
          components: rustfmt, clippy
```

**Recommended fix**

```yaml
      - uses: actions/checkout@v4

      # Honour rust-toolchain.toml — keep lockstep with outbox-dispatcher.
      - name: Install Rust toolchain
        uses: dtolnay/rust-toolchain@master
        with:
          toolchain: "1.88.0"
          components: rustfmt, clippy
```

Apply the same change (without `components` where they aren't needed) to all four jobs in `ci.yml` and the `publish` / `github-release` jobs in `release.yml`.

**Why this fix**

Pinning the toolchain in the action to the same version as `rust-toolchain.toml` makes the CI's compiler, rustfmt, and clippy versions deterministic and matched to local builds, which is the whole point of the file-based pin.

### Finding 4 — CI does not verify `.sqlx/` offline cache is in sync with the source

| Field | Value |
|-------|-------|
| **File:Line** | `.github/workflows/ci.yml:14, 18-39` |
| **Severity** | Medium |
| **Category** | Database |

**Problem**

`SQLX_OFFLINE: true` makes builds use the checked-in `.sqlx/` cache. But the workflow never asserts that the cache matches the current `sqlx::query!` macros in the source. A contributor who edits a query and forgets to run `cargo sqlx prepare --workspace` will see green CI locally and in the PR — the divergence is only noticed when someone runs against a real DB. `CLAUDE.md` makes regenerating the cache a hard requirement; CI should enforce it.

**Context** (`.github/workflows/ci.yml:10-39`)

```yaml
env:
  CARGO_TERM_COLOR: always
  RUST_BACKTRACE: 1
  # Use the checked-in .sqlx cache so all steps work without a live DATABASE_URL.

jobs:
  check:
    name: Check & Lint
    runs-on: ubuntu-latest
    steps:
      ...
      - name: cargo clippy
        run: cargo clippy --workspace --all-targets --all-features -- -D warnings
```

**Recommended fix** — add a job that runs `sqlx prepare --check` against the same Postgres testcontainers image used for integration tests, or against a service container:

```yaml
  sqlx-cache:
    name: sqlx offline cache up-to-date
    runs-on: ubuntu-latest
    needs: check
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: outbox
          POSTGRES_PASSWORD: outbox
          POSTGRES_DB: outbox_dispatcher
        ports: ["5432:5432"]
        options: >-
          --health-cmd pg_isready --health-interval 5s --health-timeout 5s --health-retries 10

    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@master
        with:
          toolchain: "1.88.0"
      - uses: Swatinem/rust-cache@v2
      - name: Install sqlx-cli
        run: cargo install sqlx-cli --no-default-features --features postgres
      - name: Apply schema fixture
        run: psql "$DATABASE_URL" -f tests/fixtures/0001_initial_schema.sql
      - name: Assert .sqlx is up-to-date
        run: cargo sqlx prepare --workspace --check
```

**Why this fix**

`cargo sqlx prepare --check` fails the build if the cache differs from what the macros require, closing the gap between "compiles with offline cache" and "matches the real schema".

### Finding 5 — `outbox-publisher`'s `sqlx` feature flag is documented as enabling re-exports but is actually a no-op

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/Cargo.toml:15-16`, `README.md:12, 91-94`, `crates/outbox-publisher/src/lib.rs:10` |
| **Severity** | Medium |
| **Category** | Cross-language (API surface) |

**Problem**

The umbrella Cargo.toml defines:

```toml
# sqlx: reserved for the Phase 2 SqlxPublisher adapter; activates no code yet.
sqlx = []
```

But the README and the umbrella crate's rustdoc both teach users to enable it:

- `README.md:12`: `outbox-publisher = { version = "0.1", features = ["derive", "sqlx", "axum"] }`
- `README.md:93`: `| sqlx | Re-exports for the SQLx adapter (use \`outbox-publisher-sqlx\` for the impl) |`
- `crates/outbox-publisher/src/lib.rs:10`: same snippet in the crate docs

Users will enable a feature that does literally nothing and assume the SqlxPublisher symbol is re-exported under `outbox_publisher::sqlx::*`. It isn't. Either ship the re-exports or remove the feature.

**Context** (`crates/outbox-publisher/Cargo.toml:12-19`)

```toml
[features]
default = []
derive = ["dep:outbox-publisher-derive"]
# sqlx: reserved for the Phase 2 SqlxPublisher adapter; activates no code yet.
sqlx = []
# axum: axum extractor for webhook verification.
axum = ["dep:axum", "dep:bytes", "dep:http", "dep:tracing"]
```

**Recommended fix** — make the feature actually re-export the adapter:

```toml
[features]
default = []
derive = ["dep:outbox-publisher-derive"]
sqlx = ["dep:outbox-publisher-sqlx"]
axum = ["dep:axum", "dep:bytes", "dep:http", "dep:tracing"]

[dependencies]
outbox-publisher-sqlx = { workspace = true, optional = true }
```

And in `crates/outbox-publisher/src/lib.rs` append:

```rust
#[cfg(feature = "sqlx")]
pub use outbox_publisher_sqlx as sqlx;
```

Then `outbox-publisher-sqlx = { version = "0.1" }` can be dropped from the README install snippet.

Alternative: if a single-crate UX is not wanted, **remove** the feature and update the README/lib.rs to stop advertising it.

**Why this fix**

A feature flag is part of the public API. Shipping a no-op feature that documentation tells users to enable is a permanent footgun — users will hit cryptic "cannot find module sqlx" errors and assume the crate is broken.

### Finding 7 — `OutboxWebhook` extractor masks malformed signature headers as missing

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/src/webhook/mod.rs:256-261` |
| **Severity** | Low |
| **Category** | Correctness |

**Problem**

When the `x-outbox-signature` header is present but contains non-ASCII / non-UTF-8 bytes, `.to_str().ok()` returns `None`. The extractor then falls back to `""`, which `WebhookVerifier::verify` reports as `MissingHeader` (400). The caller sees "missing header" even though the header *is* present, just unparseable — which is what `MalformedHeader` exists to signal. Both currently map to 400 so the HTTP response is correct, but log messages and any future telemetry will mis-classify the failure.

**Context** (`crates/outbox-publisher/src/webhook/mod.rs:253-265`)

```rust
        async fn from_request(req: Request, state: &S) -> Result<Self, Self::Rejection> {
            let verifier = WebhookVerifier::from_ref(state);

            let signature = req
                .headers()
                .get("x-outbox-signature")
                .and_then(|v| v.to_str().ok())
                .unwrap_or("")
                .to_owned();

            let body = Bytes::from_request(req, state)
                .await
                .map_err(|e| WebhookRejection(VerifyError::BodyRead(e.to_string())))?;
```

**Recommended fix**

```rust
        async fn from_request(req: Request, state: &S) -> Result<Self, Self::Rejection> {
            let verifier = WebhookVerifier::from_ref(state);

            let signature = match req.headers().get("x-outbox-signature") {
                None => return Err(WebhookRejection(VerifyError::MissingHeader)),
                Some(v) => v
                    .to_str()
                    .map_err(|_| WebhookRejection(VerifyError::MalformedHeader))?
                    .to_owned(),
            };

            let body = Bytes::from_request(req, state)
                .await
                .map_err(|e| WebhookRejection(VerifyError::BodyRead(e.to_string())))?;

            let envelope = verifier
                .verify_and_parse::<E>(&signature, &body)
                .map_err(WebhookRejection)?;

            Ok(OutboxWebhook(envelope))
        }
```

**Why this fix**

The extractor now reports the *actual* failure mode (missing vs malformed) to both the status mapping and the `warn!` log line. Status codes are unchanged because both already map to 400, so no consumer breaks.

### Finding 9 — `sleep 20` between crate publishes is fragile

| Field | Value |
|-------|-------|
| **File:Line** | `.github/workflows/release.yml:43-44, 51-52` |
| **Severity** | Low |
| **Category** | Correctness |

**Problem**

crates.io index propagation is usually <10s but can spike past a minute under load. A blind `sleep 20` will occasionally cause `cargo publish` for the downstream crate to fail with "no matching package found" until the index catches up. The job will go red and a human has to re-trigger.

**Context** (`.github/workflows/release.yml:36-47`)

```yaml
      - name: Publish outbox-publisher-derive
        run: cargo publish -p outbox-publisher-derive --no-verify
        env:
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}

      # crates.io index propagation takes a few seconds.
      - name: Wait for index propagation
        run: sleep 20

      - name: Publish outbox-publisher-sqlx
        run: cargo publish -p outbox-publisher-sqlx --no-verify
```

**Recommended fix**

```yaml
      - name: Wait for outbox-publisher-derive on index
        run: |
          for _ in $(seq 1 30); do
            if cargo search outbox-publisher-derive --limit 1 \
                 | grep -q '^outbox-publisher-derive = "0\.1\.0"'; then
              exit 0
            fi
            sleep 5
          done
          echo "outbox-publisher-derive did not propagate to the index" >&2
          exit 1
```

(repeat for each subsequent crate).

**Why this fix**

Polling the actual index turns a flaky timing race into a deterministic gate, with a generous upper bound (150s here).

### Finding 10 — Release workflow hardcodes version `0.1.0` in propagation polls

| Field | Value |
|-------|-------|
| **File:Line** | `.github/workflows/release.yml:55-65, 72-82` |
| **Severity** | High |
| **Category** | Correctness |

**Problem**

The release workflow is triggered on every `v[0-9]+.[0-9]+.[0-9]+` tag, but both `Wait for ... on index` steps grep the `cargo search` output for the literal version string `0\.1\.0`. The first time a `v0.2.0` (or any non-0.1.0) tag is pushed, both polls will exhaust their 30 × 5s loop and fail, even though propagation succeeded — the workflow will halt before publishing the umbrella crate or the SQLx adapter and the release will go out half-applied.

This regresses the spirit of the previous fix (Finding 9): the wait *is* now deterministic, but it's pinned to the version that happened to be in tree when the workflow was written.

**Context** (`.github/workflows/release.yml:50-82`)

```yaml
      - name: Publish outbox-publisher-derive
        run: cargo publish -p outbox-publisher-derive --no-verify
        env:
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}

      - name: Wait for outbox-publisher-derive on index
        run: |
          for _ in $(seq 1 30); do
            if cargo search outbox-publisher-derive --limit 1 \
                 | grep -q '^outbox-publisher-derive = "0\.1\.0"'; then
              exit 0
            fi
            sleep 5
          done
          echo "outbox-publisher-derive did not propagate to the index" >&2
          exit 1

      - name: Publish outbox-publisher
        run: cargo publish -p outbox-publisher --no-verify
        env:
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}

      - name: Wait for outbox-publisher on index
        run: |
          for _ in $(seq 1 30); do
            if cargo search outbox-publisher --limit 1 \
                 | grep -q '^outbox-publisher = "0\.1\.0"'; then
              exit 0
            fi
            sleep 5
          done
          echo "outbox-publisher did not propagate to the index" >&2
          exit 1
```

**Recommended fix**

Derive the version once from the pushed tag and reuse it in both polls. Escape the dots for grep:

```yaml
      - name: Resolve release version from tag
        id: ver
        run: |
          v="${GITHUB_REF#refs/tags/v}"
          echo "version=$v" >> "$GITHUB_OUTPUT"
          # Escape regex metachars (the dots) for use in grep.
          echo "regex=$(printf '%s' "$v" | sed 's/\./\\./g')" >> "$GITHUB_OUTPUT"

      - name: Publish outbox-publisher-derive
        run: cargo publish -p outbox-publisher-derive --no-verify
        env:
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}

      - name: Wait for outbox-publisher-derive on index
        env:
          REGEX: ${{ steps.ver.outputs.regex }}
        run: |
          for _ in $(seq 1 30); do
            if cargo search outbox-publisher-derive --limit 1 \
                 | grep -Eq "^outbox-publisher-derive = \"${REGEX}\""; then
              exit 0
            fi
            sleep 5
          done
          echo "outbox-publisher-derive did not propagate to the index" >&2
          exit 1
```

(repeat the same env-var pattern for the `outbox-publisher` wait step).

Optionally, also assert at the start of the job that `steps.ver.outputs.version` matches `cargo metadata --format-version 1 | jq -r '.workspace_default_members[0]'` or the workspace `version` so a mis-tagged release fails fast.

**Why this fix**

The release pipeline must keep working across every future tag without manual edits. Sourcing the version from `GITHUB_REF` ties the polling regex to whatever was actually tagged, removing the silent landmine that the next release would otherwise step on.

### Finding 11 — `axum-handler` example wraps `SqlxPublisher` (a Copy unit struct) in `Arc`

| Field | Value |
|-------|-------|
| **File:Line** | `examples/axum-handler.rs:18, 44, 128` |
| **Severity** | Low |
| **Category** | Idiom |

**Problem**

`SqlxPublisher` is `#[derive(Debug, Clone, Copy, Default)] pub struct SqlxPublisher;` — a zero-sized type. Wrapping a ZST in `Arc` adds an allocation and atomic refcount on every clone for *no* shared state. More importantly, this is an example: users will copy this exact pattern into their own AppState and assume the publisher requires shared ownership. It also leaks an unnecessary `use std::sync::Arc` import.

The `webhook-receiver` example correctly stores `WebhookVerifier` by value via `Clone`; `axum-handler` should mirror that.

**Context** (`examples/axum-handler.rs:18-44, 126-130`)

```rust
use std::sync::Arc;

use anyhow::{Context, Result};
use axum::{extract::State, http::StatusCode, routing::post, Json, Router};
use outbox_publisher::{event::EventContext, DomainEvent};
use outbox_publisher_sqlx::SqlxPublisher;
// ...

#[derive(Clone)]
struct AppState {
    pool: PgPool,
    publisher: Arc<SqlxPublisher>,
    webhook_url: String,
}

// ...
    let state = AppState {
        pool,
        publisher: Arc::new(SqlxPublisher::new()),
        webhook_url,
    };
```

**Recommended fix**

```rust
use anyhow::{Context, Result};
use axum::{extract::State, http::StatusCode, routing::post, Json, Router};
use outbox_publisher::{event::EventContext, DomainEvent};
use outbox_publisher_sqlx::SqlxPublisher;
// ...

#[derive(Clone)]
struct AppState {
    pool: PgPool,
    publisher: SqlxPublisher,
    webhook_url: String,
}

// ...
```

**Problem**

```
Cargo.toml.orig
src/lib.rs
```

```yaml
      - name: Publish outbox-publisher-sqlx
        run: cargo publish -p outbox-publisher-sqlx
        env:
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}
```

**Recommended fix**

```bash
DATABASE_URL=postgres://outbox:outbox@localhost:5434/outbox_dispatcher \
  cargo sqlx prepare -p outbox-publisher-sqlx -- --tests
```

Option B (fallback) — bundle the workspace cache into the published package via `Cargo.toml`:

```toml
# crates/outbox-publisher-sqlx/Cargo.toml

# … existing fields …
include = [
    "src/**/*",
    "Cargo.toml",
    "../../.sqlx/**/*",     # ship the workspace offline cache
]
```

Either option lets `cargo publish -p outbox-publisher-sqlx` complete verify with `SQLX_OFFLINE=true` set.

**Why this fix**

`cargo publish` (without `--no-verify`) reproduces a from-scratch build of the *packaged* tarball, which is what crates.io users will see. Anything the build needs at compile time — including offline query metadata — must travel inside that tarball. Per-crate caches are the documented sqlx pattern for libraries published to crates.io.

### Finding 32 — `WebhookRejection`'s `pub(crate)` inner field still hides the `VerifyError` from pattern matching

| Field | Value |
|-------|-------|
| **File:Line** | `crates/outbox-publisher/src/webhook/mod.rs:222-233` |
| **Severity** | Low |
| **Category** | Idiom |

**Problem**

Finding 20 added a `pub fn error(&self) -> &VerifyError` accessor — good for read access. But the inner field is still `pub(crate)`, which means consumers can read but cannot destructure or pattern-match: writing `let WebhookRejection(VerifyError::TimestampOutOfTolerance) = rej` doesn't compile from outside the crate because the field is private. They have to do `if let VerifyError::TimestampOutOfTolerance = rej.error()`, which is fine but inconsistent with the public tuple-struct shape suggested by `pub struct WebhookRejection(pub(crate) VerifyError)`.

Either commit to "opaque newtype" (rename to `pub struct WebhookRejection { inner: VerifyError }` with the accessor as the sole API) or commit to "transparent newtype" (`pub VerifyError`). The half-pub form is the least usable shape.

**Context** (`crates/outbox-publisher/src/webhook/mod.rs:222-233`)

```rust
/// Rejection returned when signature verification or body parsing fails.
pub struct WebhookRejection(pub(crate) VerifyError);

impl WebhookRejection {
    /// The underlying verification error.
    pub fn error(&self) -> &VerifyError {
        &self.0
    }
}
```

**Recommended fix** — pick one of these:

**Transparent (preferred for ergonomics):**

```rust
/// Rejection returned when signature verification or body parsing fails.
pub struct WebhookRejection(pub VerifyError);
```

Drop the accessor; pattern matching and direct field access work for free.

**Opaque (preferred for evolvability):**

```rust
/// Rejection returned when signature verification or body parsing fails.
pub struct WebhookRejection {
    error: VerifyError,
}

impl WebhookRejection {
    pub(crate) fn new(error: VerifyError) -> Self {
        Self { error }
    }
    /// The underlying verification error.
    pub fn error(&self) -> &VerifyError {
        &self.error
    }
}
```

Update construction sites in `from_request` from `WebhookRejection(err)` to `WebhookRejection::new(err)`.

**Why this fix**

Half-pub structs are a known footgun: the shape says "public tuple", the access rules say "private". Either shape is fine on its own. The transparent form costs nothing and gives callers full Rust pattern-matching expressiveness.

### Finding 33 — `proptest-regressions/` is shipped in the published `outbox-publisher` tarball

| Field | Value |
|-------|-------|
| **File:Line** | (workspace) `crates/outbox-publisher/proptest-regressions/webhook/signing.txt` |
| **Severity** | Low |
| **Category** | Idiom |

**Problem**

`cargo package -p outbox-publisher --list` includes `proptest-regressions/webhook/signing.txt`. This file is proptest's "saved counter-example" persistence — it's a development artefact for re-running historical failing seeds. Shipping it to crates.io accomplishes nothing for downstream users (they don't run our tests), it just inflates the package and exposes our test history.

It's tracked by git so cargo includes it; the fix is to either gitignore it or carve it out of the package include set.

**Context**

```
$ cargo package -p outbox-publisher --list --allow-dirty
…
proptest-regressions/webhook/signing.txt
src/domain_event.rs
…
```

**Recommended fix**

Two complementary changes:

1. `.gitignore` (workspace root):

```gitignore
# proptest saved-counterexample cache — local dev artefact only
proptest-regressions/
```

2. `crates/outbox-publisher/Cargo.toml`:

```toml
[package]
name = "outbox-publisher"
# … existing fields …
exclude = ["proptest-regressions/**"]
```

Either alone is sufficient; doing both belt-and-braces makes sure a future re-commit can't slip the artefact into a release.

**Why this fix**

Test-state artefacts have no place in a release tarball. `exclude` makes the package contract explicit; `.gitignore` keeps the file out of source control entirely if you don't intend to track regression seeds across the team.
